# LENS Workbench — Control Repo Setup

The setup script bootstraps a new **control repo** — your local workspace for running LENS Workbench. It clones the required dependencies, copies the IDE adapter, creates output directories, and writes configuration files so that `/onboard` can finish the job.

> **Safe to re-run.** The script pulls latest changes for repos that already exist, re-syncs `.github/`, and overwrites generated YAML files with current values.

---

## Prerequisites

| Requirement | Purpose | Required? |
|---|---|---|
| **Git 2.25+** | Cloning repos, branch resolution | Yes |
| **GitHub CLI (`gh`)** | Auto-detect username, auto-create governance repo | Recommended |
| **GitHub account** | Access to the release and governance repos | Yes |
| **PowerShell 5+** or **Bash 4+** | Running the script | One of these |

> **GitHub CLI** is optional but recommended. Without it, you must create the governance repo manually before running setup, and the wizard cannot auto-detect your GitHub username.

---

## Quick Start

### Wizard Mode (Recommended)

Run the script with **no arguments** to enter the interactive wizard:

**macOS / Linux / Git Bash:**
```bash
./setup-control-repo.sh
```

**PowerShell (Windows):**
```powershell
.\setup-control-repo.ps1
```

The wizard will:
1. Auto-detect your GitHub username (via `gh` or `git config`)
2. Walk you through each setting with smart defaults
3. Show a summary and ask for confirmation before making changes

### Parameter Mode (CI / Scripted)

Pass `--org` (bash) or `-Org` (PowerShell) to skip the wizard:

```bash
# Bash — minimal
./setup-control-repo.sh --org your-username

# Bash — custom release repo and enterprise GitHub
./setup-control-repo.sh --org your-username --release-repo my-release --base-url https://github.company.com

# Bash — dry run (preview without changes)
./setup-control-repo.sh --org your-username --dry-run
```

```powershell
# PowerShell — minimal
.\setup-control-repo.ps1 -Org your-username

# PowerShell — different orgs for release and governance
.\setup-control-repo.ps1 -ReleaseOrg release-team -GovernanceOrg governance-team

# PowerShell — dry run
.\setup-control-repo.ps1 -Org your-username -DryRun
```

## What the Script Creates

After a successful run, your control repo will look like this:

```
{control-repo}/
├── lens.core/                          ← cloned release module
│   └── _bmad/lens-work/...                     ← LENS module source
├── .github/                                    ← IDE adapter (copied from release)
│   ├── copilot-instructions.md
│   └── prompts/...
├── TargetProjects/
│   └── lens/
│       └── {governance-repo}/                  ← cloned governance repo
├── _bmad-output/
│   └── lens-work/
│       ├── governance-setup.yaml               ← governance coordinates
│       ├── initiatives/                        ← initiative state (empty)
│       └── personal/                           ← user profile (gitignored)
├── LENS_VERSION                                ← version compatibility file
└── .gitignore                                  ← updated with required entries
```

### Generated Files

| File | Purpose | Overwritten on re-run? |
|---|---|---|
| `_bmad-output/lens-work/governance-setup.yaml` | Stores governance repo path and remote URL for preflight and `/onboard` | Yes |
| `LENS_VERSION` | Version compatibility check — preflight compares this against `lifecycle.yaml` | Yes |
| `.gitignore` entries | Excludes cloned repos and personal data from version control | Appended (idempotent) |
| `.github/` | GitHub Copilot adapter files copied from the release module | Replaced |

### Generated YAML: `governance-setup.yaml`

```yaml
# Generated by setup-control-repo.sh — 2026-04-04T12:00:00Z
governance_repo_path: "TargetProjects/lens/myproject.bmad.governance"
governance_remote_url: "https://github.com/your-username/myproject.bmad.governance.git"
```

This file is read by the preflight system and `/onboard` workflow to locate your governance repo without re-prompting.

### Generated File: `LENS_VERSION`

```
3.2.0.0
```

The version is extracted from `_bmad/lens-work/lifecycle.yaml` (`schema_version` field) with `.0.0` appended. Preflight hard-stops if this doesn't match the module's expected version.

### `.gitignore` Entries

The script ensures these entries exist in your `.gitignore`:

```
_bmad-output/lens-work/personal/
.github/
lens.core/
TargetProjects/
```

## Step-by-Step Walkthrough

Here is the complete setup flow from a clean clone:

### 1. Clone your control repo

```bash
git clone https://github.com/your-username/myproject.src.git
cd myproject.src
```

### 2. Run the setup script

```bash
# The script is inside the release module (which you may not have yet).
# If lens.core/ already exists locally, run directly:
./_bmad/lens-work/scripts/setup-control-repo.sh

# If this is a brand-new repo with no release module yet,
# clone it first, then run setup:
git clone --branch beta https://github.com/your-username/lens.core.git
./_bmad/lens-work/scripts/setup-control-repo.sh
```

The wizard walks you through:

| Step | What It Asks | Default |
|---|---|---|
| 1. Control Repo Directory | Where to set up the control repo | Current git root or script location |
| 2. GitHub Account | Your GitHub org or username | Auto-detected via `gh` or `git config` |
| 3. GitHub Server | Use github.com or enterprise URL | `https://github.com` |
| 4. Release Repository | Repo name, branch, owner | `lens.core`, `beta`, your org |
| 5. Governance Repository | Repo name, branch, owner, local path | Auto-derived from control repo name |
| 6. Review & Confirm | Summary of all settings | Proceed |

### 3. Store your GitHub PAT

> **Run this in your terminal, not in AI chat.** PATs should never be typed into a chat interface.

```bash
bash _bmad/lens-work/scripts/store-github-pat.sh
```

```powershell
.\_bmad\lens-work\scripts\store-github-pat.ps1
```

### 4. Run `/onboard` in your IDE

Open VS Code with GitHub Copilot Chat and type:

```
/onboard
```

The onboard workflow will:
- Validate your authentication
- Read `governance-setup.yaml` to locate your governance repo
- Create your personal profile (`_bmad-output/lens-work/personal/profile.yaml`)
- Clone target project repos from the governance `repo-inventory.yaml`

### 5. (Optional) Install non-Copilot IDE adapters

If you use Cursor, Claude Code, or Codex in addition to GitHub Copilot:

```bash
./_bmad/lens-work/scripts/install.sh --ide cursor
./_bmad/lens-work/scripts/install.sh --all-ides
```

```powershell
.\_bmad\lens-work\scripts\install.ps1 -IDE cursor
.\_bmad\lens-work\scripts\install.ps1 -AllIDEs
```

## Governance Repo Name Derivation

The governance repo name is automatically derived from your control repo folder name:

| Control Repo Name | Derived Governance Repo |
|---|---|
| `myproject.src` | `myproject.bmad.governance` |
| `myproject` | `myproject.governance` |
| `my-app.src` | `my-app.bmad.governance` |

The `.src` suffix is replaced with `.bmad`, then `.governance` is appended. You can override this in the wizard or via `--governance-repo` / `-GovernanceRepo`.

## Re-running the Script

The script is idempotent and safe to re-run at any time:

- **Release module**: `git pull` if already cloned, fresh clone if not
- **`.github/`**: Deleted and re-copied from release module
- **Governance repo**: `git pull` if already cloned, clone (or create) if not
- **`governance-setup.yaml`**: Overwritten with current values
- **`LENS_VERSION`**: Overwritten from current `lifecycle.yaml`
- **`.gitignore`**: Entries appended only if not already present
- **Output directories**: Created if missing, left alone if present

This is useful after updating the release module branch or changing governance coordinates.

## Troubleshooting

### `gh: command not found`

Install the [GitHub CLI](https://cli.github.com/). If you can't install it:
- Create the governance repo manually on GitHub before running setup
- The wizard will still work but cannot auto-detect your username — type it manually

### Governance repo creation fails

The script uses `gh repo create` to auto-create missing governance repos as private repositories. This requires:
- `gh auth login` completed successfully
- Your account has permission to create repos in the target org

If creation fails, create the repo manually and re-run the script.

### Wrong branch / branch not found

The script automatically falls back to the remote's default branch if the requested branch doesn't exist. You'll see a warning like:

```
[WARN] owner/repo does not have branch 'beta'. Using default branch 'main' instead.
```

### `LENS_VERSION` mismatch after update

If preflight reports a version mismatch after pulling a new release module:

```
/lens-upgrade
```

Or re-run the setup script — it will rewrite `LENS_VERSION` from the current `lifecycle.yaml`.

### Missing `repo-inventory.yaml` during `/onboard`

The `/onboard` workflow expects `repo-inventory.yaml` in the governance repo root. If it's missing:

1. Navigate to your governance repo
2. Create `repo-inventory.yaml` listing your target project repos:
   ```yaml
   repositories:
     - name: "my-service"
       remote_url: "https://github.com/your-org/my-service.git"
       local_path: "TargetProjects/backend/my-service"
   ```
3. Commit and push, then re-run `/onboard`

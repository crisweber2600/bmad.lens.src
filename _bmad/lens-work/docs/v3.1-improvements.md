## 1. Promote Semantics

**Problem:** `/promote` was ambiguous — sometimes creating PRs, sometimes verifying them. The happy path auto-creates PRs via `auto_advance_promote`, so `/promote` rarely needs to create anything.

**Solution:** `/promote` is now strictly an approval-verification action.

```yaml
# lifecycle.yaml
promote:
  behavior: approval-only
  actions:
    - verify_pr_merged
    - run_sensing_check
    - transition_milestone
  auto_create_fallback: true  # Create PR only if somehow missing
```

**Impact:**
- Happy path: phases auto-advance → auto-create PRs → user approves PR → runs `/promote` to confirm
- Edge case: if PR doesn't exist, `/promote` creates it (fallback)
- Mental model: "phases auto-flow, gates auto-create PRs, I just approve and confirm"

**Files changed:**
- `lifecycle.yaml` — new `promote` section

## 2. Branch Cleanup

**Problem:** Full lifecycle creates 5 branches per feature initiative. After promotion, source milestone branches become stale.

**Solution:** Automatic squash-merge and deletion of source branches after promotion.

```yaml
# lifecycle.yaml
branch_cleanup:
  strategy: squash-and-delete
  behavior:
    post_merge: delete-source-branch
    squash_commits: true
    preserve_dev_ready: true
    preserve_initiative_root: true
  exceptions:
    - branch: "{initiative-root}"
    - branch: "{initiative-root}-dev-ready"
```

**Impact:**
- After `techplan → devproposal` promotion merges, the `{root}-techplan` branch is deleted
- `dev-ready` accumulates all history as the single clean baseline
- Initiative root branch is always preserved
- Reduces long-lived branch count from 5 to 2 (root + dev-ready)

**Files changed:**
- `lifecycle.yaml` — new `branch_cleanup` section

## 3. Hotfix-Express Track

**Problem:** Even the `hotfix` track requires a constitution gate PR, which may bottleneck urgent patches.

**Solution:** New `hotfix-express` track with informational-only gates.

```yaml
# lifecycle.yaml
tracks:
  hotfix-express:
    phases: [techplan]
    milestones: [techplan, dev-ready]
    gate_mode: informational
    bypass_constitution_gate: true
    requires_constitution_permission: true
```

**Impact:**
- Gates produce advisory reports but do not block promotion
- Must be explicitly allowed in the constitution (`permit_hotfix_express` capability)
- Otherwise identical to `hotfix` track
- No constitution can be bypassed without explicit opt-in

**Constitution example:**
```yaml
# constitution.md
capabilities:
  permit_hotfix_express: true
```

**Files changed:**
- `lifecycle.yaml` — new track in `tracks` section
- `lifecycle.yaml` — `permit_hotfix_express` in constitution capabilities

## 4. Parallel Phase Execution

**Problem:** Phases are strictly sequential. UX design (businessplan) and architecture (techplan) could overlap after the PRD is committed, reducing wall-clock time.

**Solution:** Constitution-controlled parallel phase execution within the same milestone.

```yaml
# lifecycle.yaml
parallel_phases:
  enabled: false                    # Opt-in via constitution
  constitution_controlled: true
  allowed_parallels:
    techplan_milestone:
      - phases: [businessplan, techplan]
        dependency: prd
        description: "UX and architecture proceed in parallel after PRD"
  completion_rule: all-complete
```

**Impact:**
- Disabled by default — no behavioral change unless constitution enables it
- Only the `businessplan + techplan` parallel is currently defined
- PRD must be committed before techplan can start (dependency gate)
- Milestone promotion requires ALL parallel phases to complete
- Constitution enables via `enable_parallel_phases` capability

**Files changed:**
- `lifecycle.yaml` — new `parallel_phases` section
- `lifecycle.yaml` — `enable_parallel_phases` in constitution capabilities

## 5. Artifact Validation Hooks

**Problem:** Artifact quality is only checked at adversarial review (promotion time), which catches issues late.

**Solution:** Per-artifact structural validation that runs before commit acceptance.

```yaml
# lifecycle.yaml
artifact_validation:
  enabled: true
  mode: pre-commit
  constitution_controlled: true
  validators:
    prd:
      required_sections: [Overview, Requirements, Non-Functional Requirements, Success Metrics]
      min_word_count: 500
      must_reference: []
    architecture:
      required_sections: [Overview, System Design, Data Model, API Design]
      must_reference: [prd]
      min_word_count: 500
    stories:
      each_story_requires: [Acceptance Criteria, Story Points]
      must_reference: [epics]
    # ... (see lifecycle.yaml for full list)
```

**Impact:**
- Validates required sections, word counts, cross-references before commit
- Constitution can add/override validators per domain
- Three severity levels: FAIL (blocks), WARN (advisory), PASS
- Stories individually validated for acceptance criteria and story points
- YAML artifacts (sprint-status) validated for structure and required fields

**Files changed:**
- `lifecycle.yaml` — new `artifact_validation` section
- `workflows/includes/artifact-validator.md` — updated with Steps 4-5
- `lifecycle.yaml` — `artifact_validator_overrides` in constitution capabilities

## 6. Content-Aware Sensing

**Problem:** Sensing only compares scope names (domain/service/feature), producing many false positives at domain level and missing true conflicts.

**Solution:** Content-based artifact comparison between overlapping initiatives.

```yaml
# lifecycle.yaml
sensing:
  scope_overlap:
    same_feature: high
    same_service: medium
    same_domain: low
  content_overlap:
    enabled: true
    artifacts_to_compare: [epics, stories, architecture]
    similarity_threshold: 0.7
    diff_strategy: section-headers
```

**Impact:**
- Compares actual artifact content via section heading + content block comparison
- Two initiatives with overlapping epics/stories produce a stronger signal than just sharing a service
- Similarity above 0.7 triggers HIGH CONTENT OVERLAP warning
- Reduces false positives (domain-level alerts without real conflict)
- Constitution can set `content_sensing_mode` to `hard-gate` for critical domains

**Files changed:**
- `lifecycle.yaml` — new `sensing` section (replaces implicit behavior)
- `workflows/includes/sensing-report.md` — added content-aware template

## 7. Dashboard Command

**Problem:** Seeing all initiatives requires repeatedly running `/switch` + `/status`.

**Solution:** New `/dashboard` (DB) command with Mermaid visualization.

```
/dashboard
```

**Output:**
- Mermaid diagram of all active initiatives grouped by domain/service
- Summary table with initiative, track, phase, milestone, blocking PRs, sensing alerts
- Recommendations for stale PRs, ready-to-promote initiatives, idle initiatives

**Impact:**
- Read-only command — no state changes
- Groups by domain → service hierarchy
- Detects bloking PRs (> 48h without review)
- Surfaces sensing conflicts across all initiatives
- Filters closed initiatives by default

**Files changed:**
- `module-help.csv` — new DB command row
- `workflows/utility/dashboard/workflow.md` — new workflow
- `workflows/utility/dashboard/SKILL.md` — new skill definition
- `lifecycle.yaml` — new `dashboard` section

## 8. Artifact Templates

**Problem:** Each agent produces artifacts from scratch with no structural consistency guarantee.

**Solution:** Starter templates that agents load as scaffolds.

```yaml
# lifecycle.yaml
artifact_templates:
  enabled: true
  location: assets/templates/
  constitution_controlled: true
  templates:
    product-brief: product-brief-template.md
    prd: prd-template.md
    architecture: architecture-template.md
    # ... (see lifecycle.yaml for full list)
```

**Templates provided:**
| Template | Key Sections |
|----------|-------------|
| `product-brief-template.md` | Vision, Target Audience, Success Criteria |
| `prd-template.md` | Overview, Requirements, NFRs, Success Metrics |
| `ux-design-template.md` | User Flows, Component Specifications |
| `architecture-template.md` | Overview, System Design, Data Model, API Design |
| `epics-template.md` | Epic List, Dependency Graph, Priority Matrix |
| `stories-template.md` | Story List, Acceptance Criteria, Velocity Estimate |
| `implementation-readiness-template.md` | Readiness Assessment, Risk Assessment |
| `sprint-status-template.yaml` | Sprint structure, story tracking schema |

**Impact:**
- Agents load templates as starting scaffolds, not blank pages
- Consistent artifact structure across initiatives
- Adversarial reviewers know exactly where to look
- Constitution can override templates per domain (`template_overrides` capability)
- Template coverage reported in artifact validation (advisory, not blocking)

**Files changed:**
- `lifecycle.yaml` — new `artifact_templates` section
- `assets/templates/` — 8 new template files
- `lifecycle.yaml` — `template_overrides` in constitution capabilities

## 9. Dev-Complete Milestone

**Problem:** No formal connection between dev-ready status and initiative closure. Dev execution happens outside the lifecycle.

**Solution:** Optional `dev-complete` milestone that tracks implementation completion.

```yaml
# lifecycle.yaml
milestones:
  dev-complete:
    role: "Execution complete"
    description: "All planned epics merged to develop in target repos"
    phases: []
    entry_gate: dev-complete-validation
    optional: true
    tracked_state:
      stories_terminal: true
      epics_merged_to_develop: true
      close_validation: true
```

**Impact:**
- Optional — does not change existing behavior unless used
- Tracks whether all planned stories are in terminal state (merged or deferred)
- Validates all epic branches are merged to `develop` in target repos
- `/close` with `completed` state validates dev-complete before allowing closure
- No additional PR required — state is derived from target repo branch/PR status

**Files changed:**
- `lifecycle.yaml` — new `dev-complete` milestone

## 10. Gate Collapsing

**Problem:** Even small features (≤3 stories) require 3 promotion PRs with distinct gates.

**Solution:** Constitution-controlled gate collapsing that reduces the PR chain.

```yaml
# lifecycle.yaml
gate_collapsing:
  enabled: false
  constitution_controlled: true
  rules:
    - name: small-feature
      max_stories: 3
      collapsed_gates: [adversarial-review, stakeholder-approval]
      result: single-promotion-pr
      combined_reviewers: true
    - name: medium-feature
      max_stories: 8
      collapsed_gates: [stakeholder-approval]
      result: two-promotion-prs
```

**Impact:**
- Disabled by default — constitution must explicitly enable via `collapse_gates` capability
- Small features (≤3 stories): single PR from techplan → dev-ready with all reviewers
- Medium features (≤8 stories): two PRs (techplan → devproposal → dev-ready)
- No global bypass — always constitution-controlled
- Combined reviewers: all gate reviewers appear on the collapsed PR

**Files changed:**
- `lifecycle.yaml` — new `gate_collapsing` section
- `lifecycle.yaml` — `collapse_gates` in constitution capabilities

## 11. Constitution Capabilities

Six new capabilities added to the constitution schema:

| Capability | Default | Purpose |
|-----------|---------|---------|
| `permit_hotfix_express` | false | Allow hotfix-express track |
| `enable_parallel_phases` | false | Allow parallel phase execution |
| `collapse_gates` | false | Allow gate collapsing for small features |
| `artifact_validator_overrides` | — | Override per-artifact validation rules |
| `template_overrides` | — | Override artifact templates per domain |
| `content_sensing_mode` | informational | Override content-aware sensing mode |

All capabilities follow the existing additive inheritance model:
- Lower levels add, never remove
- Resolution order: org → domain → service → repo

**Files changed:**
- `lifecycle.yaml` — expanded `constitution.capabilities` list

## 12. Migration Guide

### From v3.0 to v3.1

**Schema version change:** `schema_version: 3` → `schema_version: 3.1`

**Breaking changes:** None. All new features are additive and disabled-by-default or backward-compatible.

**Recommended adoption order:**

1. **Immediate** (no configuration needed):
   - Artifact templates — agents can start using templates immediately
   - Promote semantics — `/promote` behavior refined but backward-compatible
   - Branch cleanup — automatic, no user action needed

2. **Opt-in via constitution** (when ready):
   - Artifact validation hooks — enable when templates are in use
   - Content-aware sensing — enable for domains with initiative density
   - Gate collapsing — enable for domains with many small features

3. **As-needed**:
   - Hotfix-express track — enable only in constitutions that need it
   - Parallel phases — enable when teams are comfortable with concurrent artifacts
   - Dev-complete milestone — adopt when close-state validation is desired
   - Dashboard command — available immediately, no configuration needed

**`/lens-upgrade` compatibility:** The existing upgrade workflow will detect `schema_version: 3` and offer migration to `3.1`. No manual steps required.
